## 一、项目概述

### 核心问题
AI 写小说有三个核心痛点，本项目针对性解决：

| 痛点 | 描述 | 解决方案 |
|------|------|---------|
| **AI 发散性** | 容易跑题、偏离主线 | 三级大纲约束 + 本章禁止清单 |
| **逻辑不一致** | 人物性格、剧情前后矛盾 | 角色卡 + 伏笔追踪 + 关系图谱 + RAG |
| **文字不人性化** | 一眼看出是 AI 写的 | 文风约束 + 对话技巧 + 句式公式 |

### 第三痛点
- **张力评判缺失** — 没有机制判断故事"嗨不嗨"
- **关联回退困难** — 修改一处，相关内容无法同步更新

## 二、核心功能模块

### 2.1 写作系统

```
用户输入指令 → screenwriter 执行 → 输出文本

指令类型：
├── 续写：基于上下文继续写
├── 润色：改写得更人性化
├── 改写：用不同风格重写
└── 概括：生成章节摘要
```

### 2.2 质量控制系统

| 功能 | 说明 |
|------|------|
| **张力仪表盘** | 实时显示冲突/悬念/情感/节奏四维评分 |
| **逻辑校验** | 人物一致性、战力体系、时间线检查 |
| **伏笔追踪** | 埋入→触发→回收 全链路管理 |
| **版本回退** | 逐级快照 + 关联回退（修改一处同步所有受影响章节）|

### 2.3 知识库系统

| 模块 | 内容 |
|------|------|
| **角色卡** | 性格、语言风格、禁区、当前状态 |
| **关系图谱** | 人物关系 + 关系变更历史 |
| **事件脉络** | 时间线、关键事件、势力变化 |
| **世界观设定** | 地理、势力、战力体系、规则 |

### 2.4 数据统计

- 章节统计（总数/已完成/进行中/待写）
- 字数统计（单章/总计/目标完成度）
- 总体完成度进度条
- 预计完本时间

### 3.2 技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端** | React + TailwindCSS | 响应式 UI，组件化开发 |
| **后端** | FastAPI + Python | 高性能 API，支持异步 |
| **数据库** | PostgreSQL | 关系型数据（章节、角色、项目） |
| **缓存** | Redis | 会话、热点数据缓存 |
| **向量库** | Milvus / Pinecone | RAG 检索、语义匹配 |
| **文件存储** | S3 / MinIO | 小说文本、图片存储 |
| **AI 模型** | Claude + Kimi + DeepSeek | 分层使用，成本优化 |

### 3.3 模型分层策略

```
构思/大纲    → DeepSeek（¥1/M，性价比极高）
草稿生成    → Kimi（中文好，200K 超长上下文）
核心章节    → Claude 3.5（文字质量最高）
润色精修    → Claude 3.5
```

### 3.4 上下文管理策略

```
分层上下文：
├── LTM（长期记忆）：角色卡、世界观、战力体系
├── MTM（中期记忆）：前3章摘要、人物状态、伏笔
└── STM（短期记忆）：当前章节、写作指令

上下文预算分配（32K为例）：
├── 系统指令 ~3,000 (10%)
├── 人物档案 ~2,000 (6%)
├── 世界观设定 ~2,000 (6%)
├── 前3章摘要 ~1,500 (5%)
├── 当前章前文 ~8,000 (25%)
├── RAG检索 ~5,000 (16%)
├── 写作指令 ~500 (2%)
└── 生成空间 ~10,000 (30%)
```

## 四、团队架构

### 4.1 Agent 团队

```
┌─────────────────────────────────────────────────┐
│                   团队 Leader                    │
│              决策、审核、最终拍板                 │
└──────────────────────┬──────────────────────────┘
                       │
       ┌───────────────┼───────────────┐
       ▼               ▼               ▼
┌────────────┐  ┌────────────┐  ┌────────────┐
│screenwriter│  │  producer  │  │   artist   │
│   编剧     │  │   制作     │  │   美术     │
└────────────┘  └────────────┘  └────────────┘
```

### 4.2 各 Agent 职责

#### screenwriter（核心·写作执行者）
- 写作引擎：续写/润色/改写/概括
- Prompt 工程：角色约束、文风控制
- 质量分析：张力评判、逻辑校验、伏笔追踪
- 知识库维护：角色卡、事件脉络、关系图谱

#### producer（技术支撑）
- RAG 检索系统、向量数据库管理
- 多模型路由（Kimi/Claude/GPT）
- 版本快照与回退机制
- 关联回退追踪算法
- API 成本优化

#### artist（界面交互）
- 写作区 UI、角色/关系图谱可视化
- 统计面板（完成度、字数、章节）
- 交互体验：输入指令、实时反馈、版本对比

## 六、开发阶段

### MVP 阶段（1-2个月）
**目标**：核心写作流程跑通
- 单用户、单项目
- 基础写作（续写/润色）
- 角色卡 + 章节大纲
- PostgreSQL 存储
- 成本：< ¥500/月

### 产品化阶段（3-4个月）
**目标**：功能完整、体验流畅
- RAG 检索系统
- 多模型路由
- 版本回退 + 关联回退
- 统计面板完善
- Redis 缓存 + 向量库
- 成本：¥2000-5000/月

### 规模化阶段（5-6个月）
**目标**：稳定生产、可能引入协作
- 多模型负载均衡
- 本地开源模型（Llama/Qwen）
- 协作功能（可选）
- 成本可控

## 七、关键设计决策

### 7.1 张力评判机制
```
评判维度：
├── 冲突强度（矛盾、对抗、立场碰撞）
├── 悬念程度（伏笔、疑问、未知）
├── 情感张力（爱恨情仇、内心挣扎）
└── 节奏密度（事件推进速度）

预警触发条件：
├── 某项指标 ≤ 2/5
├── 连续 N 段张力偏低
└── 章节进度 vs 当前张力不匹配
```

### 7.2 关联回退机制
```
场景：修改关系图谱或事件脉络
↓
系统自动分析依赖关系
↓
标记所有受影响章节（需重写/需检查/自动同步）
↓
用户确认后批量回退重写
↓
知识库同步更新
```

### 7.3 文字人性化公式
```
对话三要素：
├── 不说完：省略、停顿
├── 有干扰：动作/环境打断
└── 言外意：说 ≠ 真实意思

句式变化：
├── 主语后置
├── 一长一短（呼吸感）
└── 超过3个"的"需拆句

情绪节奏：
├── 高潮：短句、戛然而止
├── 平缓：允许"发呆"式描写
└── 悲伤：慢、留白、关联记忆
```
